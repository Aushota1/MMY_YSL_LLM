1. НЕДЕТЕРМИНИРОВАННЫЙ ПОРЯДОК ТОКЕНОВ В СЛОВАРЕ
   Местоположение: BPE_STUCTUR.py, строки 181, 327
   
   Проблема:
   - Используется `current_vocab = list(vocab)` где vocab - это set()
   - Порядок элементов в set недетерминирован (может варьироваться)
   - Это приводит к разным ID для одних и тех же токенов при разных запусках
   - Нарушает воспроизводимость результатов
   
   Строка 181 (метод train()):
   ```python
   vocab = set()
   # ... заполнение vocab ...
   current_vocab = list(vocab)  # ❌ Порядок недетерминирован
   ```
   
   Строка 327 (метод continue_training()):
   ```python
   vocab = set(existing_tokens)
   # ... заполнение vocab ...
   current_vocab = list(vocab)  # ❌ Порядок недетерминирован
   ```
   
   Решение:
   - Использовать отсортированный список для детерминированного порядка
   - Или использовать OrderedDict для сохранения порядка вставки
   
   Исправление:
   ```python
   # Вместо:
   current_vocab = list(vocab)
   
   # Использовать:
   current_vocab = sorted(list(vocab))  # Детерминированный порядок
   ```

2. НЕЭФФЕКТИВНОЕ ПРИМЕНЕНИЕ СЛИЯНИЙ В ENCODE()
   Местоположение: BPE_STUCTUR.py, строки 464-483
   
   Проблема:
   - Применяем все слияния последовательно, даже если токены уже не могут быть объединены
   - На каждой итерации проходим по всем токенам
   - Это работает правильно, но неэффективно для больших словарей
   
   Текущий код:
   ```python
   for pair in self.merge_order:
       # Применяем слияние ко всем токенам
       # Даже если пара уже не встречается
   ```
   
   Решение:
   - Можно оптимизировать, проверяя наличие пары перед применением
   - Или использовать более эффективный алгоритм (например, с кэшированием)
   
   Приоритет: НИЗКИЙ (работает правильно, просто медленнее)

3. ДУБЛИРОВАНИЕ ТОКЕНОВ В _BUILD_VOCABULARY()
   Местоположение: BPE_STUCTUR.py, строки 434-438
   
   Проблема:
   - Метод итерируется по tokens, которые могут содержать дубликаты
   - Проверка `if token not in self.inverse_vocab` предотвращает дубликаты
   - Но итерация по дубликатам неэффективна
   
   Текущий код:
   ```python
   for token in tokens:  # tokens может содержать дубликаты
       if token not in self.inverse_vocab:
           # Добавляем токен
   ```
   
   Решение:
   - Использовать set для удаления дубликатов перед итерацией
   ```python
   for token in set(tokens):  # Удаляем дубликаты
       if token not in self.inverse_vocab:
           # Добавляем токен
   ```
   
   Приоритет: НИЗКИЙ (работает правильно, просто неэффективно)

4. ПРОБЛЕМА С ПРИМЕНЕНИЕМ СЛИЯНИЙ В CONTINUE_TRAINING()
   Местоположение: BPE_STUCTUR.py, строки 303-318
   
   Проблема:
   - Применяем существующие слияния к новым словам последовательно
   - Это правильно, но может быть неэффективно
   - Если токен уже полностью объединен (len(tokens) == 1), выходим
   - Но это может пропустить некоторые слияния
   
   Анализ:
   - Логика правильная: применяем слияния в порядке их создания
   - Выход при len(tokens) == 1 оптимизирует процесс
   - Проблем нет, но можно улучшить комментарии

ДЕТАЛЬНОЕ ОПИСАНИЕ ПРОБЛЕМЫ #1
================================================================================

Проблема с недетерминированным порядком токенов:

Текущий код (строка 181):
```python
vocab = set()
for word in words:
    vocab.update(word)
vocab.add('</w>')
# ...
current_vocab = list(vocab)  # ❌ Порядок недетерминирован
```

Почему это проблема:
- В Python порядок элементов в set не гарантирован (до версии 3.7)
- Даже в новых версиях порядок может варьироваться между запусками
- Это приводит к разным ID для одних и тех же токенов
- Нарушает воспроизводимость результатов обучения

Пример проблемы:
```python
# Запуск 1:
vocab = {'a', 'b', 'c'}
current_vocab = list(vocab)  # Может быть ['c', 'a', 'b']
# 'a' получает ID 5, 'b' получает ID 6

# Запуск 2 (тот же код):
vocab = {'a', 'b', 'c'}
current_vocab = list(vocab)  # Может быть ['b', 'c', 'a']
# 'a' получает ID 7, 'b' получает ID 5
```

Исправление:
```python
# Детерминированный порядок:
current_vocab = sorted(list(vocab))  # Всегда ['a', 'b', 'c']
```
