# Сравнение системы tokenier с VeriRAG (статья arXiv:2507.15664v2)

## Краткое описание систем

### VeriRAG (статья)
**Цель:** автоматический **ремонт RTL для соответствия DFT** (Design for Testability).  
Вход: Verilog с нарушениями тестируемости → выход: исправленный Verilog (синтезируемый, без DFT-ошибок, с проверкой логической эквивалентности LEC).

### Ваша система (tokenier)
**Цель:** **классификация RTL-модулей по типу** (функциональная категория: adder, spi_master, decoder, cache, alu, counter и т.д.).  
Вход: текст модуля (Verilog/SystemVerilog) → выход: метка класса (например, `"adder"`, `"spi_master"`).

---

## 1. Сходства

| Аспект | VeriRAG | tokenier |
|--------|---------|----------|
| **Домен** | RTL (Verilog), EDA/CAD | RTL (Verilog/SystemVerilog) |
| **Использование эмбеддингов** | Да: автоэнкодер выдаёт 128-мерный вектор по представлению схемы | Да: слой эмбеддингов выдаёт векторы токенов, затем пулинг → фиксированный вектор (например, 256) |
| **Векторное представление RTL** | Вектор используется для **поиска похожего** дизайна (cosine similarity) | Вектор используется для **классификации** (SVM, XGBoost, RF и т.д.) |
| **Обучение с учителем** | Энкодер обучается с метками (тип DFT-ошибки) и контрастным loss | Эмбеддинги + классификатор обучаются по классам (тип модуля) |
| **Датасет по категориям** | VeriDFT: файлы разбиты по типу DFT-ошибки (ACNCPI, CLKNPI, CDFDAT, FFCKNP) | lerning_data: файлы разбиты по типу модуля (adder, spi_master, decoder, cache, alu, counter и др.) |
| **Фиксированный вектор** | 128-мерный латентный вектор из автоэнкодера | 256-мерный (или другой) после mean/max/sequence по токенам |
| **Роль представления** | «Схожесть по структуре и типу ошибки» для RAG | «Схожесть по типу модуля» для предсказания класса |

То есть **общее**: обе системы переводят RTL в вектор и используют его в задаче, связанной с RTL; обе опираются на размеченные данные и векторизацию.

## 2. Различия

### 2.1 Задача

- **VeriRAG:** **генерация/исправление кода** (code repair). Нужно изменить исходный Verilog так, чтобы устранить DFT-нарушения и сохранить синтезируемость и логическую эквивалентность. Итог — **новый текст модуля**.
- **tokenier:** **классификация** (classification). Исходный текст не меняется; на выходе — только **метка класса** (тип модуля).

### 2.2 Входное представление RTL

- **VeriRAG:**  
  - Не сырой Verilog.  
  - Цепочка: Verilog → **Yosys** → **JSON** (топология: связи между элементами, комбинационная/последовательная логика, такты/данные).  
  - Цель — убрать имена переменных, детали реализации (LUT и т.п.) и оставить **структуру схемы**, важную для DFT.

- **tokenier:**  
  - **Сырой текст** RTL из файла (.txt).  
  - Токенизатор (BPE или словный) + слой эмбеддингов по **тексту**, без явного графа/JSON.
и
То есть VeriRAG явно использует **структурное (схемное)** представление; ваша система — **текстовое**.

### 2.3 Модель эмбеддингов

- **VeriRAG:**  
  - **Автоэнкодер** по **числовому представлению JSON**: TF-IDF по JSON → вектор 512 dim → энкодер → **128 dim**.  
  - Обучается с реконструкцией (MSE), классификацией типа DFT-ошибки (CE) и контрастным loss.  
  - Нет языковой модели и токенизации исходного кода.

- **tokenier:**  
  - **Трансформерный слой эмбеддингов** (.pth): токены исходного текста → эмбеддинги токенов (например, 256 dim), затем агрегация (mean/max/sequence и т.д.) в один вектор.  
  - Обучение, судя по коду, связано с задачей представления текста (и далее классификацией типа модуля), без реконструкции и без контрастного loss в описанном виде.

Итого: VeriRAG — **не-текстовый** автоэнкодер по структуре; tokenier — **текстовый** эмбеддинг (токены + пулинг).

### 2.4 Использование эмбеддингов

- **VeriRAG:**  
  - **Retrieval:** по косинусной близости 128-мерных векторов выбирается **наиболее похожий** эталонный дизайн из набора (35 файлов), к которому приложено **ручное исправление**.  
  - Эта пара (исходный RTL + исправленная версия) подаётся в **LLM** как few-shot пример; LLM генерирует правку для нового RTL.  
  - То есть эмбеддинги нужны **только для поиска**, а исправление делает **внешняя LLM**.

- **tokenier:**  
  - Вектор сразу подаётся в **классификатор** (SVM, XGBoost, RF, LogisticRegression, KNN и т.д.).  
  - **Нет retrieval**, нет LLM; один вектор → один класс.

### 2.5 Роль LLM и итераций

- **VeriRAG:**  
  - LLM — центральный компонент: генерирует правки кода.  
  - Итеративный цикл: компилятор + отчёты DFT → повторные запросы к LLM до успеха или отказа.  
  - Критерий успеха: синтезируемость + отсутствие DFT-нарушений + прохождение LEC.

- **tokenier:**  
  - LLM не используется.  
  - Один проход: текст → эмбеддинги → вектор → классификатор → класс. Итераций исправления нет.

### 2.6 Датасет и метки

- **VeriRAG (VeriDFT):**  
  - 437 файлов после очистки (из ~108K).  
  - Метки: **тип DFT-ошибки** (4 класса: ACNCPI, CLKNPI, CDFDAT, FFCKNP); в работе — только дизайны с **одним** типом нарушения.  
  - Разбиение: 85 train (автоэнкодер), 35 reference (для RAG), 317 test.  
  - Для reference есть **ручные исправления** и проверка (Xcelium, Cadence Conformal LEC).

- **tokenier:**  
  - Много классов (adder, spi_master, decoder, cache, alu, counter и т.д.), много файлов на класс (например, по 100).  
  - Метки: **тип модуля** (функциональная категория).  
  - Нет «исправленных» версий модулей — только текст и класс.

### 2.7 Метрики успеха

- **VeriRAG:** доля **успешно отремонтированных** RTL (синтез + нет DFT-ошибок + LEC). Улучшение в 7.72× относительно zero-shot (53.76% vs 6.96%).
- **tokenier:** точность/F1 классификации по классам модулей (accuracy, weighted F1, confusion matrix и т.д.).

## 3. Преимущества вашей системы (tokenier) по сравнению с VeriRAG

1. **Проще пайплайн:** нет LLM, нет итеративного ремонта, нет вызова EDA (компилятор, DFT, LEC). Только эмбеддинги + классификатор.
2. **Нет зависимости от EDA-инструментов:** не нужны Yosys, Xcelium, Cadence Conformal; достаточно текста и своей модели.
3. **Масштабируемость по классам:** много функциональных категорий и много примеров на класс; VeriRAG ограничен 4 типами DFT и 437 файлами.
4. **Прямая работа с исходным кодом:** не нужна конвертация Verilog→JSON; модель учится на тексте, что естественно для классификации «что за модуль».
5. **Быстрый инференс:** один проход по тексту и классификатору, без retrieval и многократных вызовов LLM.
6. **Понятная задача:** классификация типов модулей полезна для каталогизации, поиска аналогов, навигации по проекту; задача чётко определена и легко измерима (accuracy/F1).

---

## 4. Преимущества VeriRAG по сравнению с вашей системой

1. **Решает другую задачу:** именно **исправление кода** под DFT, а не только категоризация. Ваша система классы предсказывает, но не правит RTL.
2. **Структурное представление:** JSON от Yosys явно кодирует топологию и связи, что может лучше соответствовать «аппаратным» причинам DFT-ошибок, чем чистый текст.
3. **RAG + валидированные примеры:** retrieval по похожей структуре и типу ошибки + ручные исправления дают сильный контекст для LLM и дают измеримое улучшение над zero-shot.
4. **Замкнутый цикл с EDA:** проверка синтеза, DFT и LEC гарантирует, что результат не только «похож на код», но и реально корректен и тестируем.
5. **Первый специализированный DFT-датасет и бенчмарк** для LLM-based DFT repair; ваша система решает задачу классификации типов модулей, а не ремонта.

## 6. Выводы

- **Сходства:** обе системы работают с RTL, строят векторное представление и используют размеченные данные; вектор в одной системе служит для retrieval, в другой — для классификации.
- **Главное различие:** VeriRAG — это **RAG + LLM для ремонта кода под DFT** с представлением на основе **структуры схемы (JSON)**; tokenier — **классификация типа модуля по тексту** с **текстовыми эмбеддингами** и классификатором без LLM.
- Ваша система выигрывает в **простоте, скорости и независимости от EDA** и хорошо подходит для **каталогизации и распознавания типов модулей**. VeriRAG ориентирован на **автоматическое исправление DFT-нарушений** с проверкой в EDA и даёт измеримое улучшение в этой узкой задаче.
